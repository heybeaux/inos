Assistant: That's useful. A hash join on a 90-day, 1.8B-row scan with a sort step is going to be expensive on any warehouse, but a few things stand out. First, dist key on event_user_id and sort key on event_timestamp is reasonable for point lookups by user, but it's exactly wrong for a daily rollup query — your scan is sorted by time, which is good, but your JOIN is forcing a shuffle on a 1.8B-row table because campaigns and users aren't co-distributed. Before you migrate anything, can you tell me: (a) have you tried a pre-aggregated daily_events_rollup table? And (b) what does the query look like if you push the JOINs down into a CTE that filters events first?

User: Honestly, no. We have a dbt project but the growth dashboard hits the raw events table directly because the dashboard PM wanted "drilldown to event level." A pre-aggregated table is a known TODO. I think you're saying — and this is the part I want to challenge — that the actual problem might not be Redshift, it's our schema. If we fix the schema, the query gets faster, and we don't migrate at all.

Assistant: Yes — that's exactly what I'm saying. But I want to be honest about which parts of that are durable and which aren't. A pre-aggregated rollup will absolutely fix the 28s p95 for the growth dashboard. But it doesn't solve the underlying issue, which is that you're at 4 nodes of ra3.4xlarge and your events table is going to keep growing. In 12 months you'll likely be at $20k/month on Redshift just from data volume growth, and you'll be in the same conversation. Snowflake's separation of storage and compute does meaningfully help here — you can scale compute independently of storage, and you can suspend warehouses when nothing's querying. But — and this is the honest part — that benefit might be $3-5k/month, not the 50% reduction the Snowflake reps will quote you.

User: OK so let me try to summarize the decision tree as I see it. Option 1: fix the schema (pre-agg table, rebuild the rollup query, maybe revisit dist/sort keys), keep Redshift. Probably 2-3 months, includes rebuilding the dbt project, retraining the team, and a parallel-run period. Saves maybe $3-5k/month if everything goes well. Option 3: do both — fix the schema now AND plan a migration for Q3. Is there a fourth option I'm missing?

Assistant: One option you haven't mentioned: move the events table specifically to a separate engine — Iceberg on S3 with Athena or DuckDB for the dashboard query — while keeping Redshift for everything else. That's a "lakehouse" pattern. Pros: very cheap for cold/historical data, no migration of your existing Redshift workload. Cons: now you have two systems instead of one, and your team has to learn another query engine. I wouldn't recommend it for your stage unless you're really committed to a multi-engine strategy. So practically: Option 1 first is the right move. Option 2 is a decision you can defer for 6 months once you have data on whether the schema fix gave you enough headroom.

I'll write those down as the reversal conditions.
